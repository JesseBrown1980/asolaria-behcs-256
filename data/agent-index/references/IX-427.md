The NovaLUM live-device lane is now grounded enough to stop guessing and use a controlled internal-scan model.

Current strongest truths:

- the live hardware path is mediated by an `NL2 INTERFACE ADAPTER`
- Windows currently sees that path as `USB Serial Port (COM9)`
- the desktop app source uses `SerialPort.GetPortNames()` and a TCP relay client
- the sync-server source implements the paired TCP/serial protocol and encryption logic
- the current machine state does not match the preserved same-LAN topology well enough for a trusted live sync attempt
- local runtime artifacts confirm the app is built to scan for NovaLum NL2X devices by VID/PID and COM port
- Windows currently reports `COM9` settings as:
  - baud `19200`
  - parity `None`
  - data bits `8`
  - stop bits `1`
  - `RTS ON`

Strongest runtime/config artifacts:

- `D:\projects\QDD\novalink-desktop-sync-app\bin\Debug\net9.0-windows10.0.19041.0\win10-x64\appsettings.json`
- `D:\projects\QDD\novalink-desktop-sync-app\bin\Publish\logs\NL2XSync_2025-09-16.log`

What those artifacts preserve:

- the app has a `Communication.UseEncryption` configuration path
- the publish log scans for:
  - `VID_04D8&PID_000A`
  - `VID_2BDD&PID_0001`
- the same publish log explicitly records:
  - `Found device: Dispositivo Serial USB (COM9) (ID: USB\\VID_04D8&PID_000A\\5&237FFA2&0&3)`

Safest current control model:

1. prefer app-mediated or protocol-aware serial handling through the adapter
2. do not disable device encryption first
3. do not assume direct native USB control of the NovaLUM
4. treat SSID, VPN state, adapter, COM port, and desktop app state as fixture requirements

Most practical next action:

- identify whether the desktop sync app should own `COM9`
- if needed, perform a non-mutating serial/app audit before any sync or configuration change
