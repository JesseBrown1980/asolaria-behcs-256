## First exact divergence

The first exact divergence is:

- **after socket connect**
- **after `STARTING_SYNC` begins on the server**
- **before the first successful server-to-device welcome payload is observed**

In the old good path, the next visible milestone after connect is immediate server encryption detection and the welcome screen message:

- `CONNECTED TO EBACMAP ...`

In the current local path, the connection resets before any equivalent successful welcome-payload phase is preserved.

## Why this matters

This narrows the live fault domain sharply.

The current investigation no longer needs to ask whether:

- the device can be found
- the dashboard can sync
- the socket can connect
- the proxy model can ever work

All of those are already proven.

The remaining live fault domain is the early proxy/handshake stage immediately after connect:

- first server payload
- encryption/plaintext negotiation behavior
- early read/write sequencing between app and local sync-server

## Important delta to remember

Historical good path:

- remote host `charm-sync.ebacmap.com:443`
- server traffic detected as encrypted (`NL` header)
- app promoted encryption to enabled automatically

Current local path:

- local host `192.168.1.43:443`
- local sync-server was launched with `USE_ENCRYPTION=false`
- connection still resets before a successful welcome-payload phase is recorded

This encryption-mode difference is an important delta, but it is not yet proven as the sole blocker.
