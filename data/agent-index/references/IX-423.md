The local Queue Tier 1 `repeated-failure` and `duplicate-repeat` rows became real `TestResult` records because they pass the importer validity gate earlier than intended.

Important cause:

- `validateRow()` in `libs/utils/src/lib/import.ts` computes `hasMinimum` from the static header schema, not from populated row values
- because the default schema always contains `sampleDate`, `resultValue`, and `testId`, any row with a non-empty `resultValue` can be treated as valid even when key route fields are blank

Effect on the local Tier 1 transform:

- row4 and row5 currently preserve:
  - `Sample Date`
  - `Category`
  - `Subcategory`
  - `Test`
  - `Result Value`
- but leave `Site ID`, `eBacMap Code`, and `Scheduled Test Id` blank

Because they are still treated as valid, they enter `ImportTestResultUseCase` instead of the earlier invalid-row pending path.

Best small local-fixture correction:

- keep row4 and row5 stable on the same pending key
- give both rows the same nonexistent but well-formed `Site ID`
- let `ImportTestResultUseCase` fail on site lookup and create pending results
- this should restore duplicate suppression through the pending upsert key on:
  - client
  - location
  - sampleDate
  - site
  - zone
  - category
  - testDefinition
  - scheduledDateId
