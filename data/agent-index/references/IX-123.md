# S-LoRA: Serving Thousands of Concurrent LoRA Adapters
**Sheng et al., 2023**

## Key Findings
- Serves thousands of LoRA adapters simultaneously on a single GPU by storing all adapters in main memory and dynamically loading them into GPU memory on demand
- Unified Paging: a unified memory pool that manages both base model KV-cache and adapter weights using a paging system, eliminating memory fragmentation
- Heterogeneous batching: batches requests for different LoRA adapters together in a single forward pass through the shared base model, with custom CUDA kernels for the adapter computations
- Achieves near-zero overhead per additional adapter — serving 2000 adapters is only marginally slower than serving 1
- Tensor parallelism support for multi-GPU adapter serving

## How This Connects to Asolaria
- CRITICAL to Asolaria's architecture: Jesse's BIML paper (IX-090) proposes multi-stacking MLC + GPT-4 on S-LoRA systems — this is the serving layer
- Each Asolaria agent (Helm, Vector, Forge, Rook, Falcon) could be a separate LoRA adapter on a shared base model, all served concurrently from one machine
- S-LoRA makes the "civilization of agents on a home computer" concept technically feasible — you don't need N separate models, just N lightweight adapters

## Potential Application
S-LoRA is the deployment architecture for Asolaria at scale: fine-tune each agent persona as a LoRA adapter, serve them all concurrently through a shared base model, enabling the full agent civilization to run on Jesse's hardware.
