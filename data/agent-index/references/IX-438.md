The current NovaLUM stack exposes three different control layers and they must not be confused.

## 1. Software-side app controls

Confirmed writable or software-driven:

- desktop app process state
- COM port ownership/reconnect behavior
- app-side encryption mode used by the client stack
- app-side read/write timeout behavior

Best live config surface:

- `C:\Users\acer\AppData\Local\User Name\com.companyname.nl2xsyncapp\Data\appsettings.json`

Observed values:

- `Communication.UseEncryption = true`
- `ReadTimeout = 2000`
- `WriteTimeout = 2000`

Code authority:

- `D:\projects\QDD\novalink-desktop-sync-app\Services\ConfigurationService.cs`
- `D:\projects\QDD\novalink-desktop-sync-app\Services\DeviceService.cs`

Operational meaning:

- yes, software-side client settings can be changed
- but they are not the same thing as changing the device's own saved network/global state

## 2. Device-readable internal state

Confirmed readable through the app on COM9:

- serial number
- firmware version
- assay/channel/surface/location counts
- sync server address
- sync port

These values were read from device globals during the March 29 internal scan.

Operational meaning:

- the current app path is proven for reading live device state
- this is trustworthy authority for the current fixture

## 3. Lower-level device/internal commands

The device's internal menu exposes lower-level commands such as:

- set date
- set time
- write EEPROM value
- read EEPROM value
- toggle service
- toggle autorun
- toggle WiFi image upgrade

But:

- the current desktop sync path does not prove a safe, explicit UI workflow for changing those values
- write-level device mutation should not be assumed safe just because the internal menu text is visible

Operational rule:

- do not blindly mutate device-level settings or EEPROM-level values
- treat app config changes as reversible software controls
- treat device-global reads as safe observation
- treat device writes as a separate high-risk lane requiring explicit intent and a saved before/after packet

Best current interpretation:

- yes, there is real software control on the computer side
- no, that does not mean every NovaLUM/device/adapter setting should be changed from the backend without a dedicated write plan
