This is the current live significance boundary for the March 29 NovaLUM desktop-app repair pass.

What is now true:

- the old dashboard behavior is preserved in the log: `Connect USB` used to run full verification traffic (`READ_GLOBALS`, `Ctrl-C`, `#`) immediately after opening `COM9`
- the new dashboard behavior is also preserved in the same log: `Connect USB` now performs a passive USB attach only and sends no maintenance commands
- the physical device stayed on the normal main screen during passive attach, which disproves the earlier broader theory that opening USB alone necessarily traps the unit

What remains broken:

- explicit `Read Globals` still fails after passive attach
- current live log authority shows:
  - no response after binary-mode switch attempt
  - no response to `READ_GLOBALS`
  - no response to `READ_FIRMWARE`
- the desktop app now reports that failure honestly and keeps sync locked instead of faking success

Why this matters:

- the problem class moved again
- the bug is no longer "USB connect hijacks the NovaLUM"
- the bug is now narrower: the app must intentionally promote a passive USB session into the device maintenance lane before `Read Globals`
- the next likely protocol requirement is explicit maintenance wake-up (`Ctrl-C`) before `#` binary-mode switch on the explicit `Read Globals` path

Live evidence anchors:

- desktop app log: `D:\projects\QDD\novalink-desktop-sync-app\bin\Debug\net9.0-windows10.0.19041.0\win10-x64\logs\NL2XSync_2026-03-29.log`
- current app path: `D:\projects\QDD\novalink-desktop-sync-app`
- current host listener: local QDD sync-server still live on `443`

Operational rule now:

- `Connect USB` is safe for passive attach
- `Read Globals` is the remaining explicit maintenance boundary
- if the unit ever gets trapped, the intended software escape remains `Force Restart`
