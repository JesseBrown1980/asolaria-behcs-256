The current NovaLUM desktop app exposes a configuration mismatch that later agents must not guess past.

## Confirmed file-level config

`C:\Users\acer\AppData\Local\User Name\com.companyname.nl2xsyncapp\Data\appsettings.json`

Current values:

- `Communication.UseEncryption = true`
- `ReadTimeout = 2000`
- `WriteTimeout = 2000`

## Confirmed live runtime behavior

Fresh log:

- `C:\Users\acer\Downloads\logs\NL2XSync_2026-03-29.log`

Startup lines:

- `Configuration initialized with default settings - encryption detection enabled`
- `Config UseEncryption value: False`
- `Creating EncryptionService with useEncryption=False`
- `Serial communication encryption is disabled`

## Current code authority

Most relevant source files:

- `D:\projects\QDD\novalink-desktop-sync-app\MauiProgram.cs`
- `D:\projects\QDD\novalink-desktop-sync-app\Services\ConfigurationService.cs`
- `D:\projects\QDD\novalink-desktop-sync-app\Services\DeviceService.cs`

Important observed behavior:

- `MauiProgram` registers `ConfigurationService` directly and the inspected bootstrap path does not call `AddJsonFile`, `ConfigurationBuilder`, or any equivalent JSON config loader
- `ConfigurationService` initializes `_useEncryption = false`
- `DeviceService` consumes `_configService.GetUseEncryption()` at startup
- the inspected code path currently supports the stronger conclusion that `appsettings.json` is not the authoritative startup source for `UseEncryption`

## Operational meaning

- later agents must not assume that editing `appsettings.json` alone will change live encryption behavior
- the current authoritative live signal is the runtime log plus the in-memory service path
- app-side settings are still safer to change than device-global settings, but they require a mutation-ready packet and verification loop

## Practical rule

Before any NovaLUM setting mutation:

1. snapshot current `appsettings.json`
2. snapshot the fresh NL2X runtime log
3. identify whether the intended setting is file-backed, in-memory only, or device-backed
4. verify the effect by reopening the app and reading the new startup log
