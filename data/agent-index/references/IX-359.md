# QDD eBacMap Monorepo — Full System Map

Location: `D:\projects\QDD\ebacmap-master`
Version: 4.7.19 (root), managed by Turborepo + pnpm 10.32
Node: 20.x required

## The 4 Apps

### 1. web-ebacmap (v5.0.10) — Main Web Application
- **Framework:** Next.js 15.5.7 + React 18 + TypeScript 5.9
- **UI:** react-md 5/6, Tailwind CSS 4, Headless UI, Heroicons, styled-components
- **Database:** MongoDB 6.16 via Mongoose 8.15, Redis (ioredis + redis)
- **Auth:** iron-session, SAML SSO (node-saml), JWT (jose), bcrypt
- **State:** Redux + react-redux, SWR for data fetching
- **PDF:** Puppeteer 13 + pdf-lib for generation, Pug templates
- **File handling:** Filestack (upload), AWS S3 (storage), sharp (images), XLSX export
- **SFTP:** ssh2-sftp-client + ftp for lab result import/export
- **Payments:** Stripe (react-stripe-js + stripe server SDK)
- **Monitoring:** Sentry (nextjs + node + react + profiling)
- **Charts:** Chartist, react-big-calendar
- **Push:** web-push notifications
- **Maps:** photo-sphere-viewer for 360-degree facility maps/virtual tours
- **i18n:** next-intl
- **Drag & drop:** dnd-kit
- **Scheduling:** node-cron, swab schedule wizard

### 2. admin (v1.0.12) — Admin Dashboard
- **Framework:** Next.js 15.5.7 + React 18 (same stack as web-ebacmap)
- Near-identical dependency set to web-ebacmap; manages QDD superadmin operations
- Separate deployment target for internal QDD staff

### 3. queue-server (v2.0.11) — Background Job Processor
- **Runtime:** Express 5.2.1 + ESM (tsx watch in dev)
- **Jobs:** MongoDB-backed queue (MongoQueue), cron scheduling via node-cron
- **Capabilities:** FTP connection testing, CSV parsing, XLSX export, web-push, Playwright for headless browser tasks
- **Build:** esbuild
- Processes queued jobs: report generation, FTP imports, scheduled notifications

### 4. sync-server (v2.0.1) — Real-time Socket Sync
- **Runtime:** Express 5.2.1 + ESM
- **Purpose:** WebSocket/serial communication bridge for real-time data sync
- **Encryption:** Optional (USE_ENCRYPTION env flag)
- **Build:** esbuild
- Smallest app; handles live data synchronization between field devices and database

## Architecture Pattern: Clean Architecture / DDD

Each module follows strict folder convention:
```
modules/<name>/
  domain/       — Entity classes, Value Objects, enums
  repos/        — Repository implementations (MongoDB)
  services/     — Domain services
  use-cases/    — Each use case in its own folder with UseCase.ts, dtos.ts, index.ts, types.ts
  subscribers.ts — Domain event handlers
  validation-schema.ts — Yup validation schemas
```
