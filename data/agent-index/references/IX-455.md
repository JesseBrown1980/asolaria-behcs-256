The current Omnispindle-style selector work is no longer just a Wi-Fi/profile dropdown.

The backend architecture is now:

- canonical selector store: `data/selector-catalog.json`
- typed registry: `src/selectorCatalogRegistry.js`
- hardware-capacity tumbler: `src/modelLaneTumbler.js`
- backend API: `routes/catalog.js` mounted at `/api/catalog`

The resolved selector payload is also normalized for downstream consumers:

- `categoryId`
- `categoryLabel`
- `profileLabel`
- `accessLevelLabel`
- `kind`
- `executionMode`

This architecture is the correct shape for project-agnostic selectors because it separates:

- selector definition
- selector profile
- selector access level
- truthful capability metadata
- runtime availability
- hardware-capacity enforcement

## What is now required of selectors

Every selector must declare:

- `profileId`
- `accessLevel`
- `selectorKind`

Where:

- `profileId` describes the operational posture
- `accessLevel` describes who may safely apply or use it
- `selectorKind` must truthfully distinguish `reference` from `control`

## What the tumbler means

The hardware tumbler is the anti-oversubscription layer.

It behaves like a revolver with an autoloader:

- each model/worker lane must acquire a slot
- no lane may exceed its `maxActive` count
- additional attempts queue instead of starting blindly
- releasing a slot autoloads the next queued request

Current tumbler classes:

- `cpu_local_general`
- `gpu_primary_model`
- `browser_remote_surface`
- `device_lab_serial`

## Important truth boundary

The catalog is intentionally honest about authority.

Already-real control families:

- connection fixtures
- work-org selection
- worker lanes
- desktop control primitives
- WhatsApp phone targets

Reference-only until dedicated executors exist:

- AnyDesk
- ExpressVPN product control
- Remote Desktop / mstsc managed executor

Partial selector families:

- Telegram
- Google Workspace
- GCP

## Operational significance

This gives Asolaria one backend-capable category for:

- network and Wi-Fi selectors
- hosted-work identity selectors
- remote-control selectors
- messaging selectors
- cloud selectors
- worker/model selectors

And it adds the missing machine-law:

- no selector should imply power it does not have
- no worker/model lane should run past the hardware tumbler ceiling
