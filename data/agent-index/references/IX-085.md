# Server Boot Sequence (after app.listen)

Source: server.js lines 23898-23970 [CODE-VERIFIED]

## Immediate (on listen callback)
1. **Self-watchdog** — lib/selfWatchdog, 60s interval (Phase 1 process supervision)
2. **Phone health check** — src/phoneHealthCheck, 300s interval (Phase 6)

## Async (non-blocking)
3. **MQTT broker** — `ensureLocalMqttBroker()` — starts Aedes on 127.0.0.1:18883 (IX-083)
4. **MQTT bridge** — `startMqttBridge()` — connects client to broker, publishes health/colony/presence/world state on topics:
   - asolaria/test/local/runtime/health
   - asolaria/test/local/agents/colony/status
   - asolaria/test/local/runtime/presence
   - asolaria/test/local/world/state

## Deferred (900ms delay)
5. **Fast health cache** — warm up health endpoint
6. **Trusted ecosystem cache** — refresh trusted entities
7. **Civilization world panel** — build world bundle (56 nodes, 88 edges, 180min window)
8. **Super coder cockpit** — build coder panel

## Minimal Startup Mode (default: ON)
If `ASOLARIA_MINIMAL_STARTUP_MODE=true` (default):
- Skips: external observed prune scheduler, model audit scheduler, Telegram polling
- These are deferred — not run at boot

## Full Startup Mode
If minimal mode is off, also starts:
9. **External observed prune** — cleans old external data
10. **Model audit scheduler** — daily model audit
11. **Telegram polling** — Telegram bot polling loop

## Key Insight
Admin terminals (Helm/Sentinel) are NOT started in this sequence. They're started by the /api/admin-terminals/cockpit/ensure endpoint AFTER the server is healthy. The server boots → becomes healthy → THEN admin terminals can be ensured via API.
