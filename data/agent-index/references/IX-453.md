The current direct NovaLUM Wi-Fi lane is real, but it is not the same thing as the historically proven desktop-app proxy lane.

## What the current logs prove

The device itself now reaches the local sync-server directly over Wi-Fi:

- `D:\projects\QDD\ebacmap-master\reports\novalum-sync-server-20260329-175500-enc.out.log`
- `client connected ::ffff:192.168.1.10:2000`
- `socket ended`

This repeated while the device UI showed `synchronizing`.

## What the historical good path proves

The only surviving end-to-end successful NovaLUM sync artifact is the desktop-app proxy path:

## What the code and docs say

The surviving architecture documentation and code both point to the desktop app as the proven bridge:

- `C:\Users\acer\OneDrive\Desktop\ONE-PATH\week-1-ship-qdd\charm-v2-analysis\analyst-09.md`
  - `The desktop app (NovaLink) bridges USB serial to this TCP server.`
- `D:\projects\QDD\ebacmap-master\apps\sync-server\src\SyncClient.ts`
  - `init()` immediately logs `STARTING_SYNC`
  - sends `CONNECTED TO EBACMAP v...`
  - waits for an ACK before continuing to `readGlobals()`

## Practical divergence

The current direct device Wi-Fi lane reaches TCP connect, but it does not advance to the first historically proven post-connect milestone:

- no successful welcome ACK
- no first `CONNECTED TO EBACMAP` payload completion
- no sustained session

So the direct device Wi-Fi path is:

- real as a network path
- currently failing at the earliest handshake stage
- not yet a substitute for the historically proven desktop-app proxy path

## Operational rule

Do not treat these two lanes as interchangeable:

1. desktop-app proxy path
2. direct device Wi-Fi path

The current direct Wi-Fi behavior is a real signal, but the historical good baseline still belongs to the desktop-app proxy lane.
