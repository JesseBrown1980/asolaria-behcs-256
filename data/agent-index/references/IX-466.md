# IX-466 — Sovereignty Kernel Boots From USB

## What was built

Two new modules unify five previously disconnected subsystems:

### sovereignty-boot.js
Single entry point that loads from `ASOLARIA_SOVEREIGNTY_ROOT` (USB) or falls back to local:
- Colony registry schemas (WHO exists, authority levels)
- Transport epoch schemas (WHICH connections are real vs myth)
- Draft promotion policy (HOW facts promote: draft → local_canon → shared_canon)
- Selector catalog (WHAT hardware is available, ceiling enforcement)
- Tumbler state (4 classes: cpu_local_general, gpu_primary_model, browser_remote_surface, device_lab_serial)
- Index metadata (compiled truth from CATALOG.md)
- Lazy-loaded ASO kernel for typed operations

### sovereignty-gate.js
Enforces tiered promotion on all write operations:
- Tier 1 (routine): author + hash, deferred compile verify
- Tier 2 (structural): author + source kind check + snapshot requirement. Draft-only sources BLOCKED from structural ops.
- Tier 3 (firmware): HARD BLOCKED until CH341A hardware exists

## What was wired
- server.js: sovereignty-boot loaded on startup, status in /api/health/fast and /api/sovereignty/status
- aso-client.js: all 9 write operations gated through sovereignty-gate before local kernel calls
- schema.js: agentIndexRoot reads from sovereignty USB when available, falls back to local
