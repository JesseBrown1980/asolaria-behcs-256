The NovaLUM device-side configuration values are not read-only by definition.

## What is proven

The preserved protocol and source tree show a real write path for device globals:

- `D:\projects\QDD\ebacmap-master\apps\sync-server\src\models\Globals.ts`
- `D:\projects\QDD\NL2XSyncApp\Common\Constants.cs`

Specific proof:

- globals read command is `0x30`
- globals write command is `0x32`
- the serial/server model exposes writable device-global fields including:
  - `wifi_ssid`
  - `wifi_psk`
  - `wifi_StaticIpAddr`
  - `wifi_IpNetMask`
  - `wifi_IpGateway`
  - `bUseStaticSsidIpAddress`
  - `securityMode`
  - `sync_port`
  - `sync_server`
  - `wifi_DNS1`
  - `wifi_DNS2`

The older app-side tree also preserves the same protocol constant:

- `WRITE_GLOBALS = 0x32`

## What is not proven

The current live desktop UI does **not** expose a standard editable settings panel for these fields.

That means:

- the device-global write path exists at protocol level
- but using it requires a deliberate lower-level mutation path
- it should not be confused with the ordinary app UI flow

## Operational significance

This settles the control question:

- yes, device-side config values are accessible and writable in principle
- no, they should not be changed just because they are reachable

Current live evidence says the device already points to:

- `sync_server = 192.168.1.43`
- `sync_port = 443`

and the laptop now actually serves that endpoint locally.

Therefore the currently observed failure boundary is **after** the device-global settings layer:

- the sync-server accepted a client connection
- logged `STARTING_SYNC`
- then the socket reset

So changing device globals now would be premature and would not target the observed protocol-stage failure.
