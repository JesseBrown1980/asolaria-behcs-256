# IX-467 — Sovereignty Kernel Post-Prod Gaps

## What's live
IX-466 sovereignty kernel is in prod. Server boots from USB. Gate enforces tiered promotion.

## What's missing

### 1. Empty colony registry
Colony registry schemas are loaded but contain zero colony entries. Liris, Amy, Gaia, Falcon, and all sub-colonies need to be registered with scope, status, authority rules, and transport refs.

### 2. Author credential validation
The gate checks that an author string exists but does NOT validate it against the colony registry. Any string passes. The gate should verify that the author is a registered colony with write authority for the target layer.

### 3. exFAT limitations
E:\sovereignty\ is on exFAT. No Linux file permissions, no encryption, no journaling. The truth is on a filesystem designed for thumb drives. Eventually needs ext4+LUKS on the reserved G: partition, formatted from Linux.

### 4. QDD orchestration readiness
Asolaria can now orchestrate QDD through the sovereignty kernel instead of raw planning. The selector catalog has worker lanes. The tumbler has hardware ceilings. The proof ladder exists. But no QDD-specific orchestration has been wired through the kernel yet.

### 5. Duplicate dead code
server.js line 23370 has an unreachable /api/health/fast definition. Dead code from worker 1's edit.

### 6. USB sync
No automatic mechanism to sync sovereignty data changes back to USB. Manual copy required.

## Priority order
1. Register colonies (unlocks author validation)
2. Wire author validation into gate (unlocks real access control)
3. QDD orchestration through kernel (unlocks the token savings)
4. exFAT → ext4+LUKS migration (unlocks encryption)
5. Clean dead code (housekeeping)
6. USB sync automation (convenience)
