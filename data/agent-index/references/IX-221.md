# Rayssa/Liris Path Audit

## Findings

- Local docs still model Rayssa/LIRIS as a remote sub-colony/external-node path.
- I do not find local evidence that a Google-account security shell activated that path.
- The durable connection surfaces in local memory are bridge relay + sender-policy + sender tokens, MQTT, WhatsApp relay, optional SSH, and a suspect direct `4799` side channel.
- `runtime/outbound-liris-inbox.ndjson` explicitly shows Gaia falling back to WhatsApp relay until Asolaria restart plus bridge sender-policy registration.

## Live Host Check — 2026-03-22

- Main Asolaria runtime is listening on `0.0.0.0:4781` with PID `376`.
- Local MQTT is listening on `127.0.0.1:18883` with PID `16828`.
- Bridge relay `8788`, direct bridge `4799`, and planned LAN MQTT `18886` are not listening.
- `/api/health/fast` reports relay, tunnel, and notify all `running=false` with `pid=0`.

## Meaning

Treat the Rayssa path as documented but not currently live on the sovereign host.

For current archaeology and QDD coordination, the strongest explanation is bridge/MQTT/token/WhatsApp infrastructure drift, not Google-shell activation.
