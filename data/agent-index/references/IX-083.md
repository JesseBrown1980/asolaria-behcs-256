# MQTT Broker Startup — What Actually Happens

## How It Starts [CODE-VERIFIED]
server.js line 23907: `await ensureLocalMqttBroker()` runs during server boot.

Uses mqttLocalBrokerManager.js defaults:
- Host: 127.0.0.1 (DEFAULT_HOST)
- TCP: 18883
- WS: 18884
- WSS: 18885
- Override via env: ASOLARIA_MQTT_BIND_HOST

## Port 18886 — DOES NOT EXIST IN CODE
IX-072 said "Port 18886 = LAN port per Constitution." CORRECTION: the Constitution mentions it but the code does NOT implement it. Grep of entire src/ finds zero references to 18886.

The Constitution (Section 4A) says "18886 LAN" but this is [PROPOSED] not [CODE-VERIFIED]. Reality drift — documenting aspirations as truth.

## How to Actually Enable LAN Access
Set env var before Asolaria starts:
```
ASOLARIA_MQTT_BIND_HOST=0.0.0.0
```
This makes the EXISTING broker (18883) listen on all interfaces. No separate LAN port needed.

## The Two-Broker Problem (IX-069)
When we started a second broker with ASOLARIA_MQTT_BIND_HOST=0.0.0.0, the OLD broker (started by server.js on 127.0.0.1) kept running. Two brokers = messages don't cross.

## Correct Fix
Set ASOLARIA_MQTT_BIND_HOST=0.0.0.0 in the environment BEFORE starting Asolaria. Then the server's own ensureLocalMqttBroker() binds to 0.0.0.0 from the start. ONE broker. All messages cross.

## Where to Set It
- In Start-Asolaria-OneButton.cmd (add `set "ASOLARIA_MQTT_BIND_HOST=0.0.0.0"`)
- Or in Windows User environment variables (persistent)
- OneButton.ps1's Set-ConnectionEnvDefaults function is the right place
