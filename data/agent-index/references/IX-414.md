First practical recovery lane:

Queue / SFTP should be re-proved before EZ Protect.

Why:
- Queue has a tighter local recovery loop
- route trigger, queue-server code, Mongo runtime, and repaired path definitions already exist locally
- the main missing dependency is a recreated or configured FTP/SFTP import target plus one controlled file

EZ Protect should remain second:
- local code and tests are real
- but live proof still depends on deployed auth, representative data, and external integration shape

Operational rule:
- recover Queue live proof first
- treat EZ Protect as a mixed local/external recovery lane until auth/data/integration proof is re-established
