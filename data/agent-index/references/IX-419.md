Agents that run tests should not start from an opaque generic "test mode."

They must declare a test profile up front so the proof target is known before any environment is touched.

Recommended profiles:

- `localhost-proof`
  - localhost DB and services only
  - synthetic or seeded-local fixtures
  - no real device required
- `lan-device-proof`
  - real device present
  - SSID, VPN state, and USB/Wi-Fi mode must be explicit
  - same-LAN visibility may matter
- `staging-proof`
  - real remote environment
  - real auth/data assumptions
- `prod-proof`
  - bounded verification only
  - explicit approval and rollback/checklist required

Required test knobs:

- `profile`
- `app_env`
- `db_target`
- `network_mode`
- `vpn_state`
- `device_mode`
- `fixture_mode`
- `proof_target`

QDD defaults:

- Queue re-proof -> `localhost-proof`
- NovaLUM live sync -> `lan-device-proof`

Guardrail:

An agent should refuse to proceed when the requested proof lane is topology-sensitive and the SSID/VPN/device mode are still unknown.
