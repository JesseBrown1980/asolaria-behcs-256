The March 29 owner-saved startup bundle is actionable, but only if it is treated as a bounded recovery packet instead of a pile of duplicate artifacts.

## What the bundle actually is

- the three `amy*-asolaria-source-attach-20260329-182212.zip` files are byte-identical and should be treated as one canonical Asolaria source snapshot
- `amy-oracle-codex-bootstrap-20260329-133940.zip` is the Windows bootstrap pack for `Amy-Oracle`
- `amy-rose-codex-bootstrap-20260329-133940.zip` is the macOS bootstrap pack for `Amy-Rose`
- the duplicated WhatsApp screenshots are evidence, not separate startup steps

## Durable startup order

1. restore one local Asolaria source tree from the canonical source snapshot
2. run the host-specific Codex bootstrap pack on the target host
3. start `codex` and use the generated bootstrap prompt
4. start the Asolaria runtime surfaces from the restored source tree
5. verify `4781` and `4791` health before claiming startup success

## Why the source tree matters

The preserved console screenshot shows the scaffold/index step succeeded, but the remaining blocker on that host was still:

- no local Asolaria source tree
- operator needed to provide a repo path or approved source

So the startup packet is incomplete until the source snapshot is restored into a real working tree on the target machine.

## Verification boundary

Do not treat any of the following as proof of recovery by themselves:

- a successful bootstrap prompt generation
- a screenshot showing indexing/scaffold success
- a listener merely binding to `4781`
- a healthy gateway on `4791` while audit integrity is still locked

The recovery is only real when:

- the source tree exists locally
- the intended startup command runs from that tree
- the main runtime answers health on `4781`
- the gateway is healthy and not blocked by audit-integrity lock state

## Current preserved blockers

- the crash transcript shows `4781` could bind but still time out on health routes
- the gateway on `4791` could stay up while reporting audit-integrity failure and locked authority
- the owner-saved packet therefore preserves a restart path, not proof that the runtime is already healthy
