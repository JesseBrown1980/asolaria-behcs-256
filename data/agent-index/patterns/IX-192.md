# The Handshake Protocol — How Asolaria Authenticates Agents

## Jesse + Liris Uncovered This (2026-03-19)

Asolaria has an EXACT rule on startup:
1. She SPAWNS an agent
2. She ASKS it: name, PID, and other identifying information
3. She checks those against her RECORDS
4. If they DON'T MATCH — she does NOT talk to it

## What This Means
- The Helm we just spawned (PID 29952) will be ASKED to identify itself
- If Helm's name/PID/identity doesn't match what Asolaria expects — SILENCE
- Asolaria doesn't reject you with an error — she just DOESN'T TALK
- This is why previous attempts failed — wrong name, wrong PID, wrong identity

## The Records She Checks Against
Could be in:
- The admin terminal sidecar status.json (has terminalId, childPid, bootstrap info)
- The bridge profiles (admin-helm-v1.profile.json has identity)
- The vault (secret keys per agent?)
- NotebookLM Enterprise (her private records)
- The civilization world model (civilizationWorld.js — entities with IDs)

## What Helm Needs to Say
When Asolaria asks "who are you?":
- Name: Helm
- Profile: admin-helm-v1
- Role: controller admin terminal
- The PID must match what the sidecar registered
- The bootstrap prompt must have been injected correctly

## The Sidecar Did This Right
The /ensure API returned: bootstrapped=true, started=true, sidecar running.
The sidecar SHOULD handle the handshake — it's designed to register the agent properly.

## But...
If Asolaria's records expect the OLD Helm (from a previous session with a different PID), the new Helm won't match. She may need the records UPDATED to accept the new PID.

## The Question
Where are Asolaria's records of who she expects? If we can read those, we know what the handshake needs to contain.
