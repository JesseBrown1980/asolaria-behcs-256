# GNN Runtime Pipeline — The Complete Flow

This documents the full data path from raw runtime events to predictive agent pre-spawning. Three JS modules and one Python sidecar form a closed loop.

## Pipeline Stages

```
graphRuntimeStore.js          gnnDataExport.js            Python sidecar
┌─────────────────┐     ┌─────────────────────┐     ┌──────────────────┐
│ append event     │ ──→ │ load NDJSON events   │ ──→ │ train model      │
│ emit graph_event │     │ extract node feats   │     │ (train_baseline) │
│ write NDJSON     │     │ extract edge feats   │     │ save .pt weights │
└───────┬─────────┘     │ export JSON dataset  │     └──────────────────┘
        │               └─────────────────────┘
        │
        ▼
gnnConstructionWatcher.js (always-on intelligence layer)
┌──────────────────────────────────────────────────────┐
│ Listen: graph_event, agent_despawned, agent_completed│
│ Build: _routingGraph (source → target, confidence)   │
│ Build: _patternMemory (keyword → constructionId)     │
│ Predict: predictConstruction() → best target         │
│ Pre-spawn: queuePreSpawn() → up to 4 queued agents   │
└──────────────────────────────────────────────────────┘
```

## Stage 1: graphRuntimeStore.js — Event Capture

- **File:** `src/graphRuntimeStore.js`
- **Core:** `graphRuntimeEmitter` (EventEmitter, maxListeners=64) + NDJSON append to `graph-runtime-events.ndjson`
- Normalizes every event with: actor/target node refs (type, id, label, domain, criticality), risk scoring via `edgeRiskEngine.scoreEdgeRisk()`, timestamps
- Emits `"graph_event"` on every append — this is the live firehose that downstream consumers tap
- Also writes manifests to `graph-runtime-manifests.ndjson`

## Stage 2: gnnDataExport.js — Export for Training

- **File:** `src/gnnDataExport.js`
- Reads `graph-runtime-events.ndjson` (via `loadEvents()`, optional limit)
- `extractNodeFeatures()`: builds 6-dim node vectors — edgeVolume, avgRisk, maxRisk, failureRate, online (always 1), trustTier (runtime=3, local_client=2, else=1)
- `extractEdgeFeatures()`: builds labeled edges — risk >= 6 OR status blocked/denied = suspicious (label=1), else benign (label=0)
- Outputs JSON with `{ nodes[], edges[], stats{} }` — the format the Python trainer expects

## Stage 3: Python Sidecar — Model Training

- **Location:** `services/gnn-sidecar/`
- `train_baseline.py` consumes the exported JSON, trains EdgeLevelGNN (or other models per IX-354)
- Temporal 70/15/15 split, Adam optimizer, BCELoss, 50 epochs default
- Saves `baseline_model.pt`
- Shadow mode fallback: if PyTorch missing, runs heuristic evaluation instead

## Stage 4: gnnConstructionWatcher.js — Predictive Routing

- **File:** `src/gnnConstructionWatcher.js` (287 lines)
- **Always-on** — `startWatcher()` hooks into three event sources simultaneously:

### Event Sources

1. **`graphRuntimeEmitter.on("graph_event")`** — learns from construction processing events and energy transfers
   - `category === "construction_processing"` → extracts keywords (action, constructionId, actor, target) → feeds `recordPatternHit()`
   - `action === "construction_energy_transfer"` → records routing edge from actor to target via `recordRoutingEdge()`

2. **`constructionEmitter.on("agent_despawned")`** — triggers predictive pre-spawning
   - When an agent despawns from a construction, calls `predictConstruction("", fromId)`
   - If confidence > 0.5, queues the predicted target for pre-spawning

3. **`spawnerEmitter.on("agent_completed")` + `spawnerEmitter.on("tracer_observation")`** — learns from agent outputs and tracer detections
   - Completed agent output text → keyword extraction → pattern memory
   - Tracer-detected patterns → pattern memory

### Intelligence Structures

- **`_routingGraph`:** `Map<sourceId, Map<targetId, { count, confidence, lastAt }>>` — frequency-based routing confidence. Confidence = edge count / total outbound from source. Updated on every energy transfer.
- **`_patternMemory`:** `Map<keyword, { constructionId, hitCount, lastAt }>` — maps keywords to constructions. Updated from agent output, tracer observations, and construction events.
- **`_preSpawnQueue`:** Array, max 4 entries. FIFO with shift on overflow. Consumed by `popPreSpawn()`.

### Prediction Algorithm (`predictConstruction`)

Two strategies, combined:
1. **Pattern memory:** tokenize message into words >= 3 chars, find best keyword match by hitCount. Needs hitCount >= 2.
2. **Routing graph:** if current construction known, find highest-confidence outbound edge.
3. **Decision:** routing graph wins if confidence > 0.6, else fall back to pattern memory.

Returns `{ constructionId, confidence, source: "routing_graph"|"pattern_memory" }` or null.

## The Closed Loop

The system is self-reinforcing:
1. Agents interact → graph events recorded → GNN trains on historical patterns
2. Agent despawns → watcher predicts next target → pre-spawns agent there
3. Pre-spawned agent handles message faster → its output feeds back into pattern memory
4. More data → better predictions → faster routing → better construction pipeline

## Diagnostics

`getWatcherStatus()` returns: active flag, top-20 routing edges by confidence, pattern memory size, top-20 patterns by hitCount, pre-spawn queue contents.

## Mistakes to Avoid

1. **Do not call `stopWatcher()` casually** — it does `graphRuntimeEmitter.removeAllListeners("graph_event")`, which kills ALL graph_event listeners, not just the watcher's. This could break other consumers.
2. **Pre-spawn queue is only 4 deep.** If predictions pile up, oldest gets dropped (FIFO shift).
3. **Training is manual.** The watcher learns routing patterns at runtime in-memory, but the actual GNN model training (`train_baseline.py`) must be triggered manually. The watcher's prediction algorithm is rule-based (frequency + keyword), not the trained neural network. The GNN model is used by Sentinel for threat scoring, not by the watcher for routing.
4. **Pattern memory has no decay.** Old patterns never expire. Over time, stale keywords could dominate. Consider adding time-decay if the system runs long.
