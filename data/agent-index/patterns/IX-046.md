# Simulation is a VIEWER, Not the Process

From Dan Edens (2026-03-18), correcting our assumptions:

## What Dan Actually Does
- The simulation/3D/SwarmDesk stuff is **static data pulled from his regular git workflow**
- It's NOT running inside the same process as his actual work
- It reads a **database fed by an entirely disconnected system**
- There's no human waiting between steps — no delay from visualization
- He has an actual product that ISN'T the simulation

## The Real Secret Sauce
**Logging and audit trailing** — not the 3D view.
The simulation is just ONE of several viewers for the same data.

## What This Means for Asolaria's IX System
Our IX chain-linked index IS the data layer (like Dan's database).
The visualization (Code City, graph-runtime, Civilization Map) should be VIEWERS that READ the index — not part of the process.

```
[Agent does work] → [Logs to IX entries] → [CATALOG.md = the database]
                                                    ↓
                                          [Viewer 1: Code City 3D]
                                          [Viewer 2: graph-runtime.html]
                                          [Viewer 3: SwarmDesk]
                                          [Viewer 4: CLI catalog search]
```

## Jesse's Question: "Is it me or the agent that doesn't understand?"
Neither is wrong. Jesse IS creating a linking system using paths and constructs. The distinction Dan makes is: the INDEX is the source of truth, the SIMULATION is one way to view it. They're connected but decoupled. The IX entries we write → feed the visualization. The visualization doesn't slow down the work.
