# IX-365 — Absorbable Patterns from AI Healthcare into Asolaria

## 1. GNN Architectures (HIGH VALUE — already partly in Asolaria)
**Source:** backend/anomaly_detector/models/ (4 architectures)
- GSL-GNN (96.66%) uses attention-based graph structure learning — learns adjacency from node features
- Contrastive GNN (94.71%) uses supervised contrastive loss for anomaly separation
- Prototype GNN (94.24%) uses prototype-based learning for few-shot scenarios
- All use torch_geometric GCNConv with graceful fallback to plain nn.Linear when torch_geometric not installed
- **Asolaria already has EdgeLevelGNN in simulation (IX-322, IX-354).** These 4 architectures are the SAME models. Verify if AI Healthcare versions are newer or identical before absorbing.
- Clinical graph builder adds patient-level graph construction on top of the generic GNN layer

## 2. OCR Pipeline (NOVEL — Asolaria has nothing like this)
**Source:** backend/ocr/
- Dual-engine approach: Tesseract for printed text, EasyOCR for handwritten/complex
- Medical parser: regex-based extraction of lab values (name, value, unit, reference range), medications (name, dosage, frequency, route), vital signs, conditions, dates
- FHIR mapper: converts parsed medical entities into FHIR-compliant resources
- Each extraction carries a confidence score
- **Absorption opportunity:** Phone capture → OCR → structured data pipeline. Could process images captured on Jesse's phone into indexed knowledge.

## 3. Enterprise Test Patterns (HIGH VALUE)
**Source:** tests/ (~60+ files)
- Separate test categories: unit, integration, e2e, compliance, security, cloud
- Compliance-specific tests: test_enterprise_compliance.py, test_compliance_phase4.py, test_compliance_policies.py
- Security-specific tests: test_consent_api.py, test_consent_flow.py, test_crypto_shredding.py, test_data_deletion.py
- Uses conftest.py + conftest_integration.py pattern for fixture separation
- Fixture directories: tests/fixtures/, tests/data/, tests/mocks/
- **Absorption opportunity:** Asolaria's contract tests (tests/contracts/) could adopt the fixture/mock separation pattern and compliance test categories.

## 4. Crypto-Shredding / Compliance Framework (MEDIUM VALUE)
**Source:** backend/compliance_service.py, backend/audit/, compliance_keys.json
- Per-entity encryption keys stored in JSON — when you delete an entity, you delete its key, making all encrypted data for that entity permanently unrecoverable
- Daily audit logs in JSONL format (audit-logs/audit_YYYY-MM-DD.jsonl)
- SOC2 control mapping in docs/compliance/
- HIPAA-ready privacy controls with consent API
- **Absorption opportunity:** Asolaria's audit logging and privacy controls could use this pattern for player data in the simulation.

## 5. React + Vite + TypeScript Frontend Skeleton (LOW VALUE — unfinished)
**Source:** react_frontend/src/
- Structure: components/, pages/, api/, hooks/, context/, store/, router/, i18n/, utils/, styles/
- Vite + TypeScript + assets pipeline
- Tests in __tests__/
- **Warning:** README explicitly states "no API calls wired up, several components are stubs." The Streamlit frontend is the only working UI.
- **Absorption opportunity:** The folder structure is a clean React template but nothing novel beyond what Asolaria already has in phone-mode UI.

## 6. Fake Dataset Generation (MEDIUM VALUE)
**Source:** tests/data/, tests/fixtures/
- Synthetic patient data for testing FHIR flows
- Fake compliance/audit data
- SQL injection test strings in compliance_keys.json (e.g., "1; DROP TABLE patients;--", "1' OR '1'='1")
- **Absorption opportunity:** Pattern for generating synthetic test data for Asolaria's contract tests.

## 7. Multi-Frontend Architecture Pattern (REFERENCE)
- Streamlit (Python, working) → Desktop (PyWebview wrapper) → React (skeleton) → Mobile (Expo/RN)
- All point at same FastAPI backend on port 8000
- Desktop wrapper just opens a webview to the Streamlit port
- Mobile uses Expo with secure-store for JWT tokens + push notifications via expo-notifications
- **Lesson:** Start with Streamlit for rapid prototyping, wrap in desktop shell for distribution, build proper React frontend only when needed.
