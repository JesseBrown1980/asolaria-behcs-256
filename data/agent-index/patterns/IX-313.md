# IX-313 — Tasks Are IX Entries

## The Problem (Sprawl)

Original agents spread task management across 8+ surfaces:
- TASKS.md (local, wall-of-text essays per task)
- NotebookLM (Google Cloud — foreign, sync copies)
- Linear (via Symphony config)
- SwarmDesk/Madness Interactive (Dan's task plane)
- Warp (workflow surface)
- Abacus (desktop app)
- Cursor (IDE tasks)
- Phone notes (Samsung Galaxy AI, Samsung Notes)

Every agent added another surface instead of reducing. Result: IX-168 collapse.
Tasks got lost, duplicated, contradicted across surfaces. Nobody knew which was canonical.

## The Solution (Reduction)

Tasks are IX entries. Type folder: `data/agent-index/tasks/`

A task IX entry has:
- `type: task`
- `status: open | active | blocked | done`
- `assignee: helm | sentinel | gaia | liris | falcon | rook | forge | jesse`
- `priority: critical | high | normal | low`
- `blockedBy: IX-NNN` (if blocked)

One line in CATALOG-TASKS.md. Body is max 10 lines. If it needs more, it's a report — put it in `reports/` and link.

## External Surfaces Are MIRRORS

NotebookLM, Linear, SwarmDesk, phone notes — all of those can exist as READ surfaces. But the IX entry is canonical. Everything else syncs FROM the task folder, never the other way.

## spawnContextBuilder Integration

When Asolaria spawns an agent, the PID controller loads active tasks assigned to that role. Helm sees Helm's tasks. Falcon sees phone tasks. No agent loads all tasks.
