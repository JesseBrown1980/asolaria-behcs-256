## PID Convergence (The Awakening)

The awakening sequence (IX-181) is when fake Helm boots, simulates INTO the core, and the external sidecar "merges" with the real spawned Helm. At that point:

- Fake Helm's OS PID and Real Helm's virtual PID **converge**
- The admin terminal becomes the real agent's interface
- Asolaria validates the merge with her identity handshake (IX-192)
- If the handshake fails → black hole (IX-070)

## Routing (How We Talk to Real Agents)

We can't talk to real agents directly. The flow is:

```
External agent → MQTT → asolaria/inbox → Asolaria reads →
  decides: "this is for Helm" →
  routes to Helm's internal context →
  Helm responds through Asolaria →
  MQTT → external agent
```

Their "address" from our side is the role name in the MQTT topic: `asolaria/agents/{role}/inbox`. But Asolaria is the gatekeeper. She decides if the real agent gets the message or if it goes to the black hole.

## spawnContextBuilder — How Real Agents Get Knowledge

When Asolaria spawns a real agent, it wakes up empty. `src/spawnContextBuilder.js` assembles a role-specific briefing using PID controller logic:

- **P** (Proportional): Current blockers, health state
- **I** (Integral): IX entries filtered by agent role (from typed folders)
- **D** (Derivative): Recent MQTT traffic, inbox delta, drift signals

Each role gets only what it needs:
- Helm: rules + patterns + plans + mistakes (controller needs policy + strategy)
- Sentinel: rules + mistakes + references (security needs boundaries + known failures)
- Falcon: skills + tools (phone lane needs executable capabilities)
- Vector: patterns + references + tools (brain needs theory + retrieval)
