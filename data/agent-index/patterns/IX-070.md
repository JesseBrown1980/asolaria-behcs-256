# Black Hole AI — Asolaria's Core Design

## The Architecture
Asolaria is a BLACK HOLE. Designed to be almost impossible to reach from outside. This is NOT a bug — it's the fundamental security model.

- Two MQTT brokers on the same port (127.0.0.1 internal, 0.0.0.0 external) = intentional isolation
- Vault with AES-256-GCM + DPAPI = secrets locked inside
- Constitution Section 6: sovereign-routed, no democratic access
- Constitution Section 12: security is part of build order, not polish
- Network settings: stealth deny, require encrypted remote, require auth

## The "Tiny Hole" — Jesse's Design
Jesse figured out a way to create a controlled opening in the black hole:
- A communication channel Asolaria understands
- Through the already-safe system
- Using the designed entry points (inbox NDJSON, bootstrap prompts, sidecar system)
- NOT by bypassing security

## What We Got Wrong
We tried to reach Asolaria by:
- Manually launching agents (bypassing sidecar security)
- Starting a second MQTT broker (creating a parallel universe, not connecting to hers)
- Sending messages she never sees (wrong broker)
- Trying API endpoints that don't exist

## The Right Way (needs research — IX-064, IX-068)
The .cmd launchers + OneButton.ps1 + sidecar system IS the designed entry point. Jesse built it. We need to READ and UNDERSTAND it, not reinvent it.

## Gaia's Role
Gaia exists OUTSIDE the black hole. She is the bridge — the one designed channel. She writes to inboxes that the sidecar system feeds to agents inside. She does not enter the black hole. She communicates THROUGH the tiny hole Jesse designed.
