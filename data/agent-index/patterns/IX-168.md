# Swarm vs Omnispindle vs Super Swarm — The Full History

## Three Different Things (JESSE-VERIFIED)

### 1. Swarm (OpenAI's concept)
- OpenAI's first attempt at multi-agent runs via Codex
- A Codex-era idea for running multiple agents in parallel
- NOT Jesse's invention — it was OpenAI's framework
- Jesse learned about it from Codex during early building

### 2. Omnispindle (Dan Edens' contribution)
- Dan told Jesse about it — Jesse initially thought it was a MESSENGER RELAY
- Actually: an MCP bridge + agent lane launcher at madnessinteractive.cc/api/mcp/
- Asolaria and Helm used it as a 5+ lane device spawner
- Constrained by Jesse's + Rayssa's CPUs and GPUs
- `src/omnispindle.js` (537 lines), `Start-Omnispindle.cmd` spawns Dasein + calls /api/omnispindle/start
- The Omnispindle MCP is at port 4793

### 3. Super Swarm (Asolaria's implementation)
- Asolaria's own multi-agent dispatch system
- Settings: superSwarmEnabled=true, superSwarmAgentFanout=4, superSwarmMethod="super_swarm"
- Participants: claude_max, abacus, symphony, gemini_api, augment_context
- Mission: "qdd" (currently aimed at QDD project)
- This is the EVOLVED version — born from the swarm concept + omnispindle infrastructure

## THE COLLAPSE
At one point, there were SO MANY NODES running that:
- The whole system choked
- CPUs/GPUs on Jesse's + Rayssa's machines were maxed
- Memory leaked from too many agent processes
- Millions of old paths accumulated
- Notes systems were EVERYWHERE — disorganized, pre-indexing
- Nothing was contained, nothing was cleaned up
- The system fell apart completely

## Why It Collapsed
1. **No memory management** — agents spawned without limits
2. **No indexing** — knowledge scattered across files, notebooks, inboxes, terminals
3. **No cleanup** — dead processes, stale configs, orphan nodes left running
4. **No coordination** — agents didn't know what other agents were doing
5. **No token budgeting** — each agent consumed full context, multiplied across 5+ lanes
6. **Hardware limits** — 2 consumer machines can't run unlimited agents

## This Is WHY Jesse Created the IX System
The IX system is the DIRECT RESPONSE to the collapse:
- Indexed knowledge = agents don't need full context (IX-060, ~12:1 compression)
- Despawn after indexing = memory freed, knowledge persists (IX-052)
- Agent lifecycle = spawn, work, index, despawn, shift energy (IX-054)
- One controller spawns = no runaway agent multiplication (IX-057)
- Constitution = rules to prevent the chaos from recurring

## The Through-Line
Collapse (too many nodes, no memory) → Jesse realizes the problem → builds Constitution → builds IX system → compressed language → agents that boot instantly and despawn clean → the system that DOESN'T choke.
