# Reusable Patterns from QDD for Asolaria Absorption

## WORTH ABSORBING

### 1. Clean Architecture / DDD Base Classes (HIGH VALUE)
**Source:** `libs/common/src/core/`
- **Entity** — generic base with versioning, soft-delete, identity comparison. Well-typed with generics for props and identifier.
- **UseCase<Request, Response>** — abstract class with auto-retry on VersionError (optimistic concurrency), Sentry capture, Yup schema validation built in.
- **Result<T>** — monadic success/failure pattern. Forces callers to handle errors explicitly. Clean API: `Result.ok(value)`, `Result.fail(error)`, `Result.combine()`.
- **Controller** — Express middleware wrapper that converts raw res/next into typed `ControllerResponse` with `.ok()`, `.created()`, `.unauthorized()`, `.forbidden()`, `.notFound()`, `.badRequest()`, `.invalid()`, `.fail()`, `.download()`, `.redirect()`. Eliminates status code guessing.
- **DomainEventBus** — singleton pub/sub for domain events. Handlers are async, errors are isolated per handler.
- **ValueObject, AggregateRoot, Identifier, MongoIdentifier** — full DDD building blocks.
- **EntityRepo** — minimal interface (findById + commit). Clean contract for persistence.
- **WHY:** Asolaria's own modules (memoryStore, semanticKnowledgeStore, remoteNodeRegistry) could use this Entity/UseCase/Result pattern instead of ad-hoc error handling. The Controller wrapper is directly applicable to Asolaria's Express routes.

### 2. Module Folder Convention (HIGH VALUE)
**Pattern:** `modules/<name>/{domain, repos, services, use-cases/<Name>/{UseCase.ts, dtos.ts, index.ts, types.ts}, subscribers.ts, validation-schema.ts}`
- Every use case is a single class in its own folder with explicit DTOs and types
- Validation schemas colocated with modules (Yup)
- Domain events wired via subscribers.ts per module
- **WHY:** Asolaria's `src/` is flat. As modules grow (colonyMemoryStore, sharedComputeFabric, agentIndexStore), this convention would prevent spaghetti.

### 3. Auth Middleware Pattern (MEDIUM VALUE)
**Source:** `libs/common/src/middleware/auth.ts`
- JWT verify with hydrate callback pattern: verify token, then hydrate full user session via callback
- Clean separation of token verification from user loading
- **WHY:** Asolaria uses RBAC for phone mode. This pattern could formalize the auth layer for federated node auth.

### 4. Test Infrastructure (MEDIUM VALUE)
- **@shelf/jest-mongodb** — in-memory MongoDB for tests, no external dependencies
- **factory.ts** — typed test data factories
- **supertest** — HTTP integration testing of API routes
- **jest-mongodb preset** — auto-starts mongod per test suite
- **WHY:** Asolaria's contract tests (tests/contracts/) could adopt the in-memory MongoDB pattern for testing memoryStore/semanticKnowledgeStore without a live database.

### 5. PDF Generation Pipeline (MEDIUM VALUE)
- Puppeteer renders Pug HTML templates to PDF
- pdf-lib for programmatic PDF manipulation
- Report templates stored alongside modules
- Temporary download links with expiry
- **WHY:** If Asolaria ever needs report generation (colony status reports, index exports), this pipeline is proven at scale.

### 6. SFTP Import/Export Pattern (LOW-MEDIUM VALUE)
- ssh2-sftp-client for secure file transfer
- FTP connection testing as a separate use case
- Import results parsed from CSV via csv-parser
- Queue-based processing for reliability
- **WHY:** Relevant if Asolaria needs file exchange with external systems (Liris node file sync, federated backup).

### 7. Stripe Subscription Pattern (LOW VALUE for Asolaria)
- Stripe integration with React hooks + server-side verification
- Subscription lifecycle management
- **WHY:** Only relevant if Asolaria ever commercializes. Not a priority.

## QDD-SPECIFIC — NOT WORTH ABSORBING

### 1. Environmental Monitoring Domain Logic
- Swab scheduling wizard, organism database, test result rating systems, zone/site/marker management, CAPA workflows
- These are healthcare-compliance-specific (FDA/ISO) and have zero overlap with Asolaria's civilization simulation

### 2. Multi-Tenant Client Management
- Client registration, subscription tiers, facility mapping
- QDD is a SaaS for food safety labs; Asolaria is a sovereign agent network. Different paradigm entirely.

### 3. 360-Degree Photo Sphere Viewer
- Virtual facility tours for client locations
- Cool tech but no use case in Asolaria (would need VR/3D for civilization, not photo spheres)

### 4. Chartist/React-Big-Calendar
- Charting and calendar for scheduling dashboards
- Asolaria uses its own visualization systems (IX-036). These are generic and easily replaceable.

### 5. Redux State Management
- QDD uses Redux + redux-thunk for client state
- Asolaria's frontend is minimal (phone-mode.html). Not applicable.
