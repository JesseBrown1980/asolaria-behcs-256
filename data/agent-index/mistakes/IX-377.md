# What went wrong

Gaia let two NovaLUM lanes blur together for too long:

1. the historically proven desktop-app proxy path
2. the current direct device Wi-Fi path

That caused too much time to be spent treating device settings and direct Wi-Fi behavior as if they were automatically the same authority as the old good proxy run.

# What the evidence actually says

The historical successful path is explicit:

- `C:\Users\acer\Asolaria\tmp\docs\phone-downloads\NL2XSync_2025-06-12.log`
- `Sync button clicked - Starting serial-to-socket proxy service...`
- `Successfully connected to sync socket server`
- `INITIAL DETECTION: Server encryption mode: ENCRYPTED`
- `Proxy communication completed successfully`

The architecture docs also say the desktop app is the bridge:

- `C:\Users\acer\OneDrive\Desktop\ONE-PATH\week-1-ship-qdd\charm-v2-analysis\analyst-09.md`
- `The desktop app (NovaLink) bridges USB serial to this TCP server.`

The current direct device Wi-Fi lane is real, but only proves:

- device reaches the local server directly as `192.168.1.10`
- server accepts the connection
- socket ends immediately

That is not the same authority as the historically proven proxy path.

# How to avoid

For NovaLUM, keep these lanes separate at all times:

1. desktop-app proxy path
2. direct device Wi-Fi path

Rules:

- never let success or failure in one lane imply the other
- once direct Wi-Fi reaches TCP connect/end, stop editing Wi-Fi fields unless a specific config mismatch is proven
- compare the current run against the historical good milestones before changing more settings
- when the docs name the desktop app as the bridge, treat that as the canonical known-good path until a direct path is independently proven
