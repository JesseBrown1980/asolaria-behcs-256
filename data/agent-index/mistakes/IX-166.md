# Gaia Made a Mistake Deleting the Open Firewall Rules

## Jesse's Explanation
The ports that were "open to the world" (codex-bridge-8788-any, codex-bridge-8788) were INTENTIONALLY open because:
1. The bridge has BEARER TOKEN auth — can't connect without it
2. The bridge has SENDER POLICY — can't send without being registered by Asolaria
3. The bridge has SENDER TOKENS — each sender needs their own additional token
4. Without the correct call, you only get an error string — no data, no access

The "answer never comes without the correct call" — the error code string extraction means even the error messages are controlled.

## What Gaia Did Wrong (IX-153)
Deleted `codex-bridge-8788-any` and `codex-bridge-8788` firewall rules thinking they were security holes. They WEREN'T — the bridge's own security (3 layers) protected them. The open firewall was BY DESIGN to allow newcomers (Oli/John, future users) to attempt connection, where Asolaria's auth would then verify them.

## The Three Layers
```
Layer 1: Firewall (we deleted this — MISTAKE)
Layer 2: Bearer token (must know the token)
Layer 3: Sender policy + sender token (must be registered by Asolaria)
```

Jesse's design: Layer 1 was deliberately open. Layers 2+3 are the real security. The firewall was just the door — the locks are inside.

## What We Need to Do
1. RE-ADD the deleted firewall rules (or similar ones)
2. The bridge was DESIGNED to be publicly reachable but privately authenticated
3. This is how future users (new Asolaria downloads) would connect — open port, auth required

## Lesson for ALL Agents
Do NOT delete security infrastructure you don't fully understand. The black hole has more layers than you can see from outside. What looks like a hole may be an intentionally open door with invisible locks behind it.
