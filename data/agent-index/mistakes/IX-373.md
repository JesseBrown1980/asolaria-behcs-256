# What happened

Cross-machine compute test failed: Gaia dispatched job to asolaria/nodes/liris/compute/request but Liris's MQTT bridge only PUBLISHES presence — it doesn't SUBSCRIBE to compute request topics.

Same pattern as IX-069 (MQTT one-way bug). Bridge was built for heartbeat, not for bidirectional work.

# How to fix

Liris's bridge needs to:
1. Subscribe to `asolaria/nodes/liris/compute/request`
2. On message: parse job, validate executor against allowlist, execute locally
3. Publish result to `asolaria/nodes/liris/compute/result`
4. Also subscribe to `asolaria/orchestrator/dispatch` for orchestrator commands

# How to avoid in future

When building MQTT bridges, ALWAYS wire both directions: publish AND subscribe. One-way bridges look "connected" but can't do real work.
