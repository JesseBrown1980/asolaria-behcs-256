## Lessons for Asolaria

### L1: Don't Build 4 Frontends When 1 Works
The Streamlit frontend works. The React, mobile, and desktop versions are incomplete scaffolds. Pick ONE frontend, make it work, ship it. Add others only when the first is proven.

### L2: Documentation Without Implementation Is Waste
4 OAuth docs, enhancement proposals, global improvement plans — but the React frontend has zero API wiring. Write docs AFTER implementation, not before.

### L3: AI-Generated Code Needs Pruning
The Cursor + Codacy setup suggests heavy AI code generation. The kitchen-sink architecture (S-LoRA + MLC + RL + AoT + RAG + GNN + OCR in one project) is a hallmark of AI-assisted scope creep — each feature is easy to scaffold but hard to integrate and maintain.

### L4: Stalled Projects Leave Artifacts
Audit logs stop in February 2026. The project likely shifted to Asolaria work. The codebase remains as-is with no cleanup.

### L5: Same Pattern as QDD (IX-361)
Both QDD and AI Healthcare show the same pattern: ambitious scope, multiple incomplete implementations, Cursor/Antigravity-assisted development, stalled work. This is Jesse's third major project alongside Asolaria. The code has VALUE (GNN models, OCR pipeline, compliance framework) but needs extraction, not continuation.

## Recommendation
- Do NOT try to "finish" the AI Healthcare project inside Asolaria
- DO extract the valuable patterns (IX-365): GNN models, OCR pipeline, compliance framework, test patterns
