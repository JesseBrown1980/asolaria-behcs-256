# What went wrong

The first pass of the selector catalog created the right backend architecture, but the resolved selector payload still exposed only the internal field names:

- `category`
- `selectorKind`

That made the API needlessly ambiguous for future agents and UI clients that would naturally ask for:

- `categoryId`
- `categoryLabel`
- `executionMode`
- `kind`

# Why it matters

This category is supposed to be a durable Omnispindle control plane.

If the payload shape is not explicit, future agents will either:

- guess at field meaning
- write adapter code inconsistently
- or drift back into selector folklore instead of a stable backend contract

# Correction

`src/selectorCatalogRegistry.js` now decorates resolved selectors with stable aliases and labels:

- `categoryId`
- `categoryLabel`
- `profileLabel`
- `accessLevelLabel`
- `kind`
- `executionMode`

The internal fields remain, but the public payload is now safer for agents and UI surfaces.

# Rule

When creating a backend catalog intended for agent reuse, do not stop at internal correctness.
The exported shape must also be stable and unambiguous for downstream consumers.
