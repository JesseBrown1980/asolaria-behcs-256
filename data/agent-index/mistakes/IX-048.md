# Broker Bind Address Not Inherited Across Restarts

## What Happened
Started broker with ASOLARIA_MQTT_BIND_HOST=0.0.0.0 but it still listened on 127.0.0.1. The old broker process (PID 20992) was still running from BEFORE we set the env var. `ensureLocalMqttBroker()` saw it as healthy and returned without restarting.

## Fix
Must STOP the old broker first, THEN start with the new env var:
```js
stopLocalMqttBroker();  // kill old
// then with env set:
ensureLocalMqttBroker();  // start fresh
```

## Verification
Always check `netstat -an | grep 18883` after starting. If it shows `127.0.0.1:18883` instead of `0.0.0.0:18883`, the old process is still running.

## CRITICAL SAFETY RULE
When restarting/reconfiguring the MQTT broker for sub-colony access:
- NEVER kill the broker without confirming Asolaria's own MQTT client will reconnect
- Asolaria connects to 127.0.0.1:18883 — this MUST keep working
- Binding to 0.0.0.0 is SAFE because it includes 127.0.0.1
- But binding to a specific LAN IP (e.g., 192.168.1.3 only) would BREAK Asolaria's connection
- After any broker restart: verify `curl http://127.0.0.1:4781/api/health` shows `mqtt.connected: true`
- Same rule applies when registering NEW sub-colonies — their firewall rules must not block existing connections
