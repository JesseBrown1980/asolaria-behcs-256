# What went wrong

The March 29 startup bundle contained real recovery material, but it was easy to overclaim what it proved:

- three byte-identical source archives could be mistaken for three separate payloads
- duplicated screenshots could be mistaken for multiple independent checkpoints
- the console screenshot showed scaffold/index success, but not a healthy runtime

That makes it dangerously easy to say “Amy is ready to start” when the actual blockers are still:

- no local Asolaria source tree on the target host
- `4781` can bind while timing out
- `4791` can be up while audit integrity and authority state are still bad

# Why it matters

If duplicate artifacts and screenshot-only success are treated as proof, the operator can restart against the wrong state:

- wrong source assumptions
- false confidence about runtime health
- restart claims that are not backed by live `4781` / `4791` truth

# Correction

The recovery packet is now constrained to one real proof order:

1. one canonical source snapshot
2. one host-specific bootstrap packet
3. one live runtime truth check

And the packet is now stamped with explicit PID/profile metadata and mirrored into owner memory.

# Rule

Do not treat duplicate startup artifacts or scaffold-only success as proof that a host is start-ready.
Source-tree presence plus live `4781` / `4791` truth outrank screenshots and duplicate archives.
