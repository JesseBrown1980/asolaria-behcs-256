# Locked Ourselves Out of the Bridge

## What Happened
1. Rotated bridge-token.txt to a new token
2. Relay process still running with OLD token in memory
3. Can't send with NEW token (relay rejects it)
4. OLD token is now only in bridge-token-old-backup.txt
5. The 7 connected bots still work (they authenticated with the old token at connect time)
6. But NEW connections can't auth

## The Fix Options
A. Use the OLD token (from backup) to send — it still works against the running relay
B. Restart the relay via bridgectl.ps1 — it reads bridge-token.txt on start, gets the NEW token
C. Revert: copy old token back to bridge-token.txt, use it, rotate AFTER restarting relay

## Lesson
ALWAYS restart the service BEFORE rotating tokens if the service holds the token in memory. Or: rotate token → restart service → THEN try to connect.

Same pattern as IX-048 (MQTT bind address) and IX-163 (token not inherited). Running processes don't read file changes.

## Recommendation
Option B: restart relay with bridgectl.ps1. Bots will reconnect via watchdog. Brief interruption but clean state with new token.
