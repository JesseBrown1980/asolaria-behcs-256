# Stale IPs In Firewall and Configs — Silent Federation Killer

## The Mistake

Firewall rules and MQTT configs store hardcoded IPs (e.g. `192.168.1.8` for Liris). When the network changes (different router, neighbor's WiFi, DHCP reassignment), federation silently fails. No error. Just silence — which looks like the encryption/stealth behavior (IX-308) and wastes hours debugging the wrong thing.

## Known Stale IPs (as of 2026-03-23)

| Where | Old IP | Current IP | Status |
|-------|--------|-----------|--------|
| Firewall rule `MQTT Broker LAN` | 192.168.1.8 | 192.168.15.170 | STALE — blocks Liris |
| Firewall rule `Oli 192.168.1.64` | 192.168.1.64 | unknown | STALE |
| IX-093 (index entry) | 192.168.1.3 = Jesse | 192.168.15.189 | STALE |

| config/liris-agent-memory.md | 192.168.1.3 | 192.168.15.189 | STALE |

## Why It Happens

Jesse's network is not static. Different router = different subnet. IPs change. Hardcoded IPs in firewall rules and configs become invisible blockers.

## FALLBACK: If Federation Doesn't Connect

**CHECK IPS FIRST.** Before debugging encryption, TLS, broker config, or stealth behavior:

1. `ipconfig` on Jesse's machine — what's the current IP?
2. `ipconfig` on Rayssa's machine — what's the current IP?
3. `ping` between them — can they see each other?
4. Check firewall: `Get-NetFirewallRule -DisplayName 'MQTT Broker LAN' | Get-NetFirewallAddressFilter`
5. If the firewall rule has a wrong IP → update it
6. If they're on different subnets → they literally cannot talk

## The Fix (when IPs change)

```powershell
# Update MQTT firewall rule to new Liris IP
Start-Process powershell -ArgumentList "-Command netsh advfirewall firewall set rule name='MQTT Broker LAN' new remoteip=<NEW_LIRIS_IP>" -Verb RunAs
```

Or open to whole subnet: `remoteip=192.168.15.0/24`

## The Rule

IPs are NOT permanent. Every time federation fails silently, check IPs BEFORE checking encryption.
