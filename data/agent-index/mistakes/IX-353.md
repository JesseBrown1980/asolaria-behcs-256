# What could go wrong

This session created 40+ IX entries in one sitting. At this rate:
- Sub-catalogs bloat past 20 rows within 2 sessions
- Agents start loading 50+ entries instead of 12
- Context windows fill with index overhead
- We recreate the IX-168 collapse at a higher level

# Rules to prevent explosion

1. **Before creating an IX entry, ask: does one ALREADY EXIST that covers this?** Update, don't duplicate.
2. **Group related discoveries into ONE entry**, not one-per-fact. IX-320 (definitive system map) is better than 50 individual tool entries.
3. **Operational entries (skills/tools) justify their existence.** Each must answer: "can an agent USE this to DO something?"
4. **Archaeology entries should be COMPRESSED.** One era summary > 20 individual historical facts.
5. **Sub-catalog > 20 rows = mandatory era split** (IX-352 pattern).
6. **Session target: 10-15 new entries max**, not 40+. Quality over quantity.

# The meta-mistake

Creating index entries about how to create index entries is recursive bloat. This entry and IX-352 are the LAST meta-entries needed. Future work should be OPERATIONAL, not meta.
