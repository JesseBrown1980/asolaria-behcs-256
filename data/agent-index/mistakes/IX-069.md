# MQTT One-Way Communication Bug

## Symptom
- Liris connected to broker at 192.168.1.3:18883 — CONFIRMED
- Liris sees sovereign health messages (asolaria/nodes/rayssa/, asolaria/test/local/*)
- Liris subscribed to # (wildcard) — should see EVERYTHING
- Gaia publishes to asolaria/liris/commands — Liris does NOT receive
- Gaia can see her own publishes echoed back locally
- Liris's publishes also don't reach Gaia

## What Liris Found
- Her inbox folder: `C:/Users/rayss/Asolaria/data/mqtt-inbox-gaia/` has files but only her own heartbeats
- She runs a live 30s listen on # — sees sovereign messages but NOT Gaia's publishes

## Possible Causes
1. **Two broker instances** — old one on 127.0.0.1:18883 (Asolaria's built-in) + new one on 0.0.0.0:18883 (we started). They're on different listen addresses. Gaia connects to 127.0.0.1 (old), Liris connects to 192.168.1.3 (new). Messages don't cross.
2. **Client ID collision** — if two clients use same ID, broker disconnects one
3. **Retained message confusion** — old retained msgs replay, new ones don't deliver
4. **Topic ACL** — broker might have topic restrictions (unlikely — Aedes has none by default)

## Most Likely: TWO BROKERS
netstat showed BOTH 127.0.0.1:18883 AND 0.0.0.0:18883 listening. Asolaria's internal client connects to 127.0.0.1. Our new broker binds 0.0.0.0. These might be SEPARATE Aedes instances on the same port but different bind addresses.

## Fix Approach
- Stop both brokers
- Start ONE broker on 0.0.0.0:18883
- Verify only one LISTENING entry in netstat
- Reconnect Asolaria's MQTT client to the same broker
- Test bidirectional: Gaia → Liris and Liris → Gaia
