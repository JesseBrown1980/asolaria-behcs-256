## Lessons for Future QDD Work

1. **Brian is the bottleneck, not the code.** EZ Protect is 89% done. The remaining 11% is blocked on Brian's DigitalOcean access, env values, LIMS specs, and real device data. Stop writing code until Brian responds.

2. **Don't maintain parallel branches for absent clients.** Two pnpm worktrees, a recommendation doc, and a version comparison — all waiting for a decision Brian hasn't made since March 11.

3. **Clean the scheduler graveyard.** 35 open branches is technical debt. Either close them with "won't fix" or batch-merge the ones that are ready. Don't let them accumulate further.

4. **QDD ≠ Healthcare.** The client project (Charm/eBacMap/lab management) and Jesse's personal AI Healthcare project are completely separate codebases. Never confuse them.

5. **Test AFTER access, not before.** Integration tests for routes that need real env vars and DigitalOcean access are premature. Unit tests for business logic are fine. But route integration tests that mock everything are less valuable than they appear.

6. **One worktree, one branch.** The two-worktree strategy created confusion about which branch agents should target. Pick one and archive the other until a decision is needed.

7. **Swarm doesn't help QDD.** QDD is a traditional monorepo with TypeScript, Next.js, Jest. It doesn't benefit from multi-agent fan-out. It needs one focused agent doing test-driven development with Brian's feedback loop. The swarm collapse (IX-168) proved this.
