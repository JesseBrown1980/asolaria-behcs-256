# IX-461 — Amy Startup Recovery Packet With Bounded 3-Lane Attack

## Objective

Turn the owner-saved March 29 startup bundle into an executable recovery packet without overloading one lane, duplicating artifacts, or claiming startup success before the live health gates pass.

## 3-and-3 rule for this attack

### Three lanes only

1. source lane
2. bootstrap lane
3. runtime lane

### Three outputs per lane

Each lane must return:

1. ground-truth facts
2. blocking conditions
3. exact next command or handoff

## Phase order

### 1. Plan and index first

- preserve the bundle as canon before new runtime mutations
- keep the source snapshot, bootstrap packs, and screenshots deduplicated
- mirror the packet into owner memory so startup recovery survives agent despawn

### 2. Source lane

- treat the three large source zips as one artifact
- confirm the expected restore root for the target host
- prove whether the missing source-tree blocker still exists

### 3. Bootstrap lane

- validate the correct host pack (`Amy-Oracle` on Windows, `Amy-Rose` on macOS)
- extract the exact bootstrap script and prompt handoff order
- keep discovery-first and ask-first elevation rules intact

### 4. Runtime lane

- validate the live startup command against `IX-309`
- separate `4781` main-runtime truth from `4791` gateway truth
- check whether gateway audit-integrity lock would still block a “healthy” restart claim

## Execution law

- do not use more than 3 concurrent agents for this packet
- do not treat duplicate archives as independent evidence
- do not claim startup success until the source tree, bootstrap path, and live runtime health all agree

## First execution target

The highest-value immediate slice is:

1. confirm the exact startup command and launcher path already saved locally
2. confirm whether the local source tree is already sufficient or the saved snapshot is still required
3. prove the real current state of `4781` and `4791`
