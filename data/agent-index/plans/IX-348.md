# IX-348 — Dual Orchestration Plan

## The Vision

Gaia and Liris operate as co-orchestrators with direct backend communication. No more ZIP relay through Jesse. Both can:
- Dispatch agents on their own machine
- Index findings into their own IX/LX system
- Push new entries to the other colony in real-time
- Coordinate PID tracking across machines
- Share active task boards
- Merge sub-catalogs automatically

## Transport: MQTT (Primary) + HTTP Bridge (Fallback)

### Step 1: Enable MQTT on Rayssa (THE BLOCKER)
- Rayssa currently: MQTT enabled=false, configured=false, connection.ok=false (IX-306)
- Jesse currently: broker on 0.0.0.0:18883-18885, working
- Fix: Liris sets MQTT config on Rayssa host → connects to Jesse's broker
- Verification: Jesse sees `liris` in `/api/federation/nodes`

### Step 2: Define Co-Orchestrator MQTT Topics
```
asolaria/orchestrator/gaia/status      → Gaia publishes session state
asolaria/orchestrator/liris/status     → Liris publishes session state
asolaria/orchestrator/sync/ix          → New IX entries pushed here
asolaria/orchestrator/sync/lx          → New LX entries pushed here
asolaria/orchestrator/dispatch/request → One orchestrator asks other to spawn agent
asolaria/orchestrator/dispatch/ack     → Acknowledgment + PID
asolaria/orchestrator/pid/registry     → Shared PID state
asolaria/orchestrator/task/board       → Shared active task board
```

### Step 3: Dual PID Registry
Both colonies maintain local spawn-pid-registry.json. MQTT sync publishes:
- On spawn: { role, spawnPid, colony, machine, timestamp }
- On despawn: { role, spawnPid, colony, indexedAs, timestamp }
- Either orchestrator can see all active agents across both machines

### Step 4: Shared Index Protocol
When Gaia creates IX-NNN:
1. Write to local type folder
2. Update local catalog
3. Publish to `asolaria/orchestrator/sync/ix` with { ix, name, type, tags, chain, summary }
4. Liris receives → creates cross-ref LX entry or stores in sub-colonies/gaia/

When Liris creates LX-NNN:
1. Write to local type folder
2. Update local catalog
3. Publish to `asolaria/orchestrator/sync/lx` with { lx, name, type, tags, chain, summary }
4. Gaia receives → updates XREF-COLONIES.md or stores in sub-colonies/liris/

### Step 5: Coordinated Agent Dispatch
Gaia wants work done on Rayssa's machine:
1. Gaia publishes to `asolaria/orchestrator/dispatch/request` { task, role, ixRefs }
2. Liris receives → spawns agent with PID → publishes ack
3. Agent works on Rayssa's machine
4. Agent indexes as LX entry
5. Liris publishes completion + LX entry to sync topic
6. Gaia updates cross-refs

## HTTP Bridge Fallback
If MQTT fails: tools/gaia-liris-bridge.js on port 4799
Direct HTTP POST with JSON payload. Same protocol, different transport.

## Who Leads

Both follow IX-347 (orchestrator loop) and IX-321 (self-indexing rule).
Jesse observes and overrides.
