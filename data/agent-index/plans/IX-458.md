# IX-458 — Agent-Aware Plug-and-Play ebacmap / NovaLum Control Plane

## The Thesis

The future state should not be manual USB rituals and hidden repair screens. It should be a device control plane where NovaLum and ebacmap-connected surfaces become agent-friendly in the same disciplined way Asolaria made worker lanes agent-friendly:

- discoverable
- typed
- auditable
- policy-gated
- reversible

## Why This Works

We already have the necessary patterns:

- `IX-455`: selector catalog + scarce-lane control
- `IX-418`: NovaLum topology and real network/device boundaries
- `IX-321`: self-indexing rule for all discoveries and mutations
- `IX-347`: orchestrator loop discipline
- `IX-319`: typed transport and fallback map
- `IX-308`: encryption-first truth for protected lanes

The missing step is to apply those patterns directly to device/runtime control.

## Architecture

### Layer 1 — Device Registry

Each device becomes a typed runtime object with:

- physical transport state
- identity
- last known globals state
- sync target
- firmware/update state
- restart/recovery state

### Layer 2 — Capability Catalog

Agents do not get raw port access. They get scoped capabilities:

- detect
- read globals
- validate sync target
- run sync
- force restart
- firmware update
- observe-only

Each capability must carry:

- risk class
- required authority
- preconditions
- rollback or recovery expectations

### Layer 3 — Policy and Approval

Device commands split into:

- read-only automation
- bounded repair automation
- high-risk mutation requiring explicit approval

### Layer 4 — Indexed Execution

Every action records:

- actor
- lane
- device
- command path
- before/after state
- outcome
- recovery note

### Layer 5 — Operator Cockpit

The operator sees:

- device readiness
- authority state
- pending approvals
- recent indexed actions
- safe next actions
- blocked actions with reasons

## Immediate Rollout

### Slice A — Safe Registration

Turn NovaLum detection into a formal device-runtime registration path without auto-hijacking the device.

### Slice B — Typed Device Authority

Expose device actions behind selector/policy gates instead of direct unrestricted command paths.

### Slice C — Audited Automation

Allow agents to run bounded playbooks such as:

- detect and report
- read globals and validate
- force restart and await recovery
- sync only if all gates pass

## Legal / Safety Boundary

This direction is only valid inside an authorized, auditable control model:

- authorized systems only
- read-only default
- explicit approval for mutating actions
- full traceability
- no silent bypass of protected controls
- elevated review for regulated or sensitive device/data lanes

## Durable Rule

Treat this as a real implementation direction. The value is not “agents everywhere.” The value is safe, indexed, traceable device automation where availability and accountability increase together.
